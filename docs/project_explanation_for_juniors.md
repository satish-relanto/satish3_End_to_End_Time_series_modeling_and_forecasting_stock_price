# Stock ARIMA Forecasting Project: Technical Explanation For Juniors

## 1. Project Overview

This project is an end-to-end stock price forecasting system using ARIMA time series modelling.

The goal is to forecast a stock's future adjusted closing price using historical OHLCV data:

- `open`
- `high`
- `low`
- `close`
- `adjusted_close`
- `volume`

The project is not just a modelling notebook. It is structured like a deployment-ready machine learning project:

- It has reusable source code.
- It has a sample dataset.
- It trains and saves a model artifact.
- It generates reports and diagnostics.
- It exposes forecasts through an API.
- It has a dashboard.
- It has tests.
- It has Docker deployment files.


## 2. Business Problem

A financial analyst or research team may want a lightweight forecasting tool to answer questions like:

- What is the short-term expected price range for a stock?
- How does an ARIMA forecast compare with a naive last-price forecast?
- How uncertain is the forecast?
- Does the time series show autocorrelation?
- Are the transformed returns stationary?

In real finance, stock prices are noisy and often behave close to a random walk. That means ARIMA should be treated as a baseline forecasting model, not as a guaranteed trading strategy.

## 6. Why We Use Adjusted Close

For stock modelling, `adjusted_close` is usually better than raw `close`.

Raw closing price can be affected by:

- Stock splits.
- Dividends.
- Corporate actions.

Adjusted close attempts to normalize these effects. In this sample project, `adjusted_close` equals `close`, but the code supports real adjusted close data when available.

## 7. Time Series Concepts Used

### 7.1 Time Series

A time series is data ordered by time.

Example:

```text
Date        Price
2025-10-27  157.63
2025-10-28  155.42
2025-10-29  153.89
```

Unlike normal tabular data, time order matters. Yesterday's price can influence today's price.

### 7.2 Stationarity

A stationary time series has statistical properties that remain stable over time.

Roughly, this means:

- The mean is stable.
- The variance is stable.
- The autocorrelation pattern is stable.

Many classical time series models, including ARIMA, work better when the series is stationary or made stationary through differencing.

### 7.3 Why Raw Stock Prices Are Usually Non-Stationary

Stock prices often trend upward or downward and can follow a random-walk-like process.

That means raw prices usually do not have a stable mean.

So instead of applying diagnostics directly to raw prices, we often use:

- Log prices.
- Differenced log prices.
- Log returns.

### 7.4 Log Price

The project converts price into log price:

```text
log_price = log(price)
```

Why use log price?

- It stabilizes percentage-based changes.
- It makes multiplicative movements easier to model.
- Forecasts can be converted back to price using exponentiation.

### 7.5 Log Returns

Log return is calculated as:

```text
log_return_t = log(price_t) - log(price_t-1)
```

This is close to percentage return for small changes.

Log returns are often more stationary than raw stock prices.

The project uses log returns for:

- Stationarity testing.
- ACF/PACF diagnostics.

## 8. ARIMA Explanation

ARIMA stands for:

```text
AutoRegressive Integrated Moving Average
```

It is written as:

```text
ARIMA(p, d, q)
```

### 8.1 AR Term: p

The `p` value is the number of autoregressive lags.

An autoregressive model uses previous values to predict the current value.

Example:

```text
today = function(yesterday, day_before_yesterday)
```

If `p = 1`, the model uses one previous lag.

### 8.2 I Term: d

The `d` value is the number of differences.

Differencing helps remove trend and make the series more stationary.

Example first difference:

```text
difference_t = value_t - value_t-1
```

If `d = 1`, the model differences the series once.

### 8.3 MA Term: q

The `q` value is the number of moving-average error lags.

It means the model uses previous forecast errors to improve current predictions.

### 8.4 ARIMA Order Used

The default project order is:

```text
ARIMA(1, 1, 1)
```

Meaning:

- `p = 1`: use one autoregressive lag.
- `d = 1`: apply one differencing step.
- `q = 1`: use one moving-average error lag.

This is a simple baseline order. In a more advanced version, we can add automatic order selection.

## 9. ACF And PACF Explanation

### 9.1 ACF

ACF means Autocorrelation Function.

It shows correlation between a time series and its own lagged values.

Example:

```text
correlation(today, yesterday)
correlation(today, two days ago)
correlation(today, three days ago)
```

ACF helps estimate the `q` part of ARIMA.

### 9.2 PACF

PACF means Partial Autocorrelation Function.

It shows the direct relationship between a value and a lag after removing the effect of intermediate lags.

PACF helps estimate the `p` part of ARIMA.

### 9.3 Why We Apply ACF/PACF To Log Returns

Applying ACF/PACF directly to raw stock prices can be misleading because prices are often non-stationary.

So this project applies ACF/PACF to log returns:

```text
log_return = diff(log(price))
```

The output is saved here:

```text
reports/AAPL_adjusted_close_acf_pacf.csv
```

### 10.3 `src/stock_arima/preprocess.py`

This file prepares the time series.

Main functions:

```python
prepare_price_series(...)
log_price_series(...)
log_return_series(...)
stationarity_summary(...)
```

Important steps:

1. Filter selected ticker.
2. Set `date` as the index.
3. Sort by date.
4. Convert to business-day frequency.
5. Forward-fill missing business days.
6. Check there are enough observations.
7. Check prices are positive.
8. Create log prices and log returns.

Why business-day frequency matters:

Stock markets do not trade on weekends. Using business-day frequency gives the model a regular time index.

### 10.7 `src/stock_arima/forecast.py`

This file generates forecasts.

### 10.8 `src/stock_arima/evaluate.py`

This file performs walk-forward evaluation.

Walk-forward validation is important for time series.

In normal machine learning, we may randomly split rows into train and test. But for time series, random splitting can leak future information into the past.

Instead, walk-forward validation does this:

```text
Train on first 45 observations -> forecast next 3
Train on first 50 observations -> forecast next 3
Train on first 55 observations -> forecast next 3
...
```

Metrics used:

- MAE.
- RMSE.
- MAPE.

The evaluation also compares ARIMA against a naive baseline.

Naive baseline means:

```text
tomorrow's forecast = last observed price
```

This is a strong baseline for stock prices because stock prices often behave like random walks.

## 20. Important Design Decisions

### 20.1 Why Not Use A Notebook Only?

Notebooks are useful for exploration, but deployment projects need reusable code.

This project separates responsibilities into modules:

- Data loading.
- Preprocessing.
- Training.
- Evaluation.
- Forecasting.
- API.
- Dashboard.

That makes the code easier to test, deploy, and maintain.

### 20.2 Why Use FastAPI?

FastAPI is used because:

- It is simple and fast.
- It has automatic request validation.
- It creates Swagger docs automatically.
- It is production-friendly.

### 20.3 Why Use Streamlit?

Streamlit is used because:

- It is fast for building dashboards.
- It works well for data science demos.
- It needs less frontend code than React or Angular.

### 20.4 Why Use Joblib?

`joblib` is used to save and load the trained model artifact.

This means we do not need to retrain the model every time the API starts.

### 20.5 Why Include A Naive Baseline?

A forecast model should be compared against a simple baseline.

For stock prices, a strong baseline is:

```text
future price = last observed price
```

If ARIMA cannot beat this baseline, it may not be adding value.

## 21. Common Warning: ARIMA ConvergenceWarning

During walk-forward evaluation, `statsmodels` may show:

```text
ConvergenceWarning: Maximum Likelihood optimization failed to converge.
```

This means the optimizer had difficulty fitting ARIMA parameters for one or more rolling windows.

This can happen because:

- Financial series are noisy.
- Some rolling windows may be hard to fit.
- The selected ARIMA order may not be ideal for every window.
- The sample window may be too small.

In the latest run, evaluation completed successfully despite the warnings.

How to reduce this later:

- Try different ARIMA orders.
- Increase initial training size.
- Add automatic order selection.
- Use SARIMAX or other models.
- Model returns instead of log prices.

## 22. Limitations

This project is a strong educational and deployment-ready baseline, but it has limitations:

- Synthetic sample data is not real market data.
- ARIMA may not capture sudden market events.
- ARIMA does not understand news, earnings, macro events, or sentiment.
- Stock prices often behave like random walks.
- Forecast intervals can widen quickly.
- This is not a trading recommendation system.

## 26. One-Minute Summary

This project demonstrates how to build a complete ARIMA time series forecasting system for stock prices.

It starts from OHLCV data, preprocesses it into a clean business-day price series, uses log transformations for modelling, applies stationarity and ACF/PACF diagnostics on log returns, trains an ARIMA model, evaluates it with walk-forward validation, saves artifacts, serves forecasts through FastAPI, and visualizes results in Streamlit.

The key engineering lesson is that a good data science project should not stop at model training. It should include data validation, diagnostics, evaluation, API serving, dashboarding, tests, documentation, and deployment files.
